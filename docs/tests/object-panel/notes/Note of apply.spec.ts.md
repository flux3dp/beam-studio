# Note of apply.spec.ts

被測檔案：`packages/core/src/web/app/components/dialogs/autoFit/apply.ts`（自動對位套用：縮放/旋轉主件，並依輪廓將圖檔複製到各工件位置）

## 對應測試表項目

對應「自動對位（2.3.9 新增）是否正常」這一列，特別是步驟 6–7：

- 「調整物件位置後套用」→ 主件的縮放、旋轉與位移。
- 「確認圖檔是否有複製到所有選擇的工件位置上」→ 針對每一個非主輪廓進行複製並定位、旋轉。

## 測試內容 — 逐案說明

- **主件縮放＋旋轉＋位移**：以 `setSvgElemSize('width', 60)` 與 `('height', 40)` 設定尺寸、`setRotationAngle(element, 30, …)` 設定角度；位移 dx/dy 以手算常數驗證（將 cos30、sin30 帶入影像中心與初始中心的偏移，得 dx≈55.98、dy≈172.32）。
- **只有主輪廓時不複製**：不呼叫 `cloneElements`、`multiSelect([element])`，也不寫入 history。
- **複製到非主輪廓（無角度差）**：`cloneElements` 呼叫 1 次，dx/dy 等於輪廓位移 (30, 40)；options 帶有 `callChangOnMove: false`、`selectElement: false`、`parentCmd`；原件與複製件會一起被選取。
- **旋轉差套用到複製件**：第二個輪廓旋轉 +90°（π/2），偏移量繞輪廓中心旋轉後得 dx≈-70、dy≈40；複製件的新角度約為 90°。
- **主件非第一個輪廓（mainIdx 非首）**：主件放在 `contours[1]`，複製件送往 index 0 與 2，dx/dy 逐一手算驗證（-80,-70 與 60,-30）。
- **角度正規化 >180°**：dAngle 為 270° 時，270-360 = -90，複製件角度約為 -90°。
- **空批次不寫 history**：尺寸命令為空且 batch 為空時，不呼叫 `addSubCommand`，也不呼叫 `addCommandToHistory`。

## 設計理由 — 為何適合單元測試

- 這是純幾何運算加上命令組裝的邏輯，最容易出錯的地方在於「三角函數符號、dx/dy 的加減、角度正規化邊界」，天生適合以手算常數釘死的單元測試。
- 經 mutation review 後確認本 spec 已具備以下重點：
  - **以手算常數取代「用同一條公式回算」**：避免測試與實作抄用同一份錯誤公式而同時出錯（若改用公式回算，符號翻轉的 mutation 會躲過檢查）。
  - **旋轉符號**：+90° 的偏移旋轉結果 (-70, 40) 方向明確，可攔截 sin/cos 符號翻轉。
  - **(-180, 180] 正規化**：270°→-90° 的邊界情境，可攔截把 `>180` 改成 `>=180` 或漏減 360 的錯誤。
  - **dx/dy 加減方向**：輪廓位移與元素偏移的組合中，主件非第一個輪廓的情境能讓「用錯基準輪廓」的錯誤無所遁形。
  - **history 記錄**：空批次不寫入 history，可攔截「即使沒有改動也塞入一筆 undo 記錄」的錯誤。

## 充分性分析

- 幾何與命令組裝的正確性已足夠（statements 100%、branches 93.75%）。
- **需 E2E 或人工驗證的部分**：套用後畫布上的視覺結果是否真的貼齊工件，以及複製件是否落在正確的實體工件上。這牽涉相機辨識出的輪廓中心是否對應真實工件，屬機器與人工驗證範疇（見 autoFit note）。單元測試這端的輪廓資料是造出來的，只驗證「給定輪廓 → 幾何算對」。

## 待確認問題（Open Questions）

- (-180, 180] 的角度正規化是**產品需求**還是**附帶的實作行為**？若下游（旋轉顯示或輸出）對 180° 邊界的處理有特定期待，應在需求層面釘住，而非僅靠本檔行為推斷。留給人工確認。
- 多選複製時原件與所有複製件會一起被選取（`multiSelect(newElements)`），這是否為使用者期望的最終選取狀態？屬 UX 決策，留給人工確認。
