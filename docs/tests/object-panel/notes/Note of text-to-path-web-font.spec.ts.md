# Note of text-to-path-web-font.spec.ts

## 對應測試表項目

右側物件面板分類：「文字轉路徑 — 可以選擇 Web 字型並正確轉路徑（例如 Noto Sans、Fira Sans）；輸入文字後，複製文字 > 原地貼上 > 轉路徑；確認轉完路徑的結果是否跟顯示相同」。與既有的 `text-to-path.spec.ts`（基本轉換）互補，邏輯層另有 Jest 的 `convertToPath.spec.ts`。

## 測試了什麼

共 3 個測試（客戶端運算，CI 可跑）：

1. **Noto Sans 轉路徑**：套用字型並確認生效、等 `document.fonts.load` 確保字型真正載入、記錄顯示中文字的 bbox，轉路徑後驗證產生非空 `d` 的真實 path，且 path 的 bbox 與顯示文字在水平方向吻合、垂直方向落在文字行框內。
2. **Fira Sans 轉路徑**：相同流程換第二個 web 字型，證明不是單一字型的巧合。
3. **複製 > 原地貼上 > 轉路徑**（表列點名的回歸場景）：複製後原地貼上，先驗證複本與原件四邊完全重疊（零位移），再對複本轉路徑，驗證結果仍與顯示吻合（畫布上剩 1 文字 + 1 路徑）。

## 設計理由

- 「結果跟顯示相同」在沒有像素比對工具下，以 **bbox 容差比對**為平台安全代理；不用 md5（跨平台字型渲染不同）。
- 垂直方向刻意只驗「path 落在文字行框內」而非相等：SVG `<text>` 的 bbox 是完整行框（ascent–descent），轉出的 path 是字形實墨範圍，直接比高度會是假失敗。
- 複製與原地貼上走鍵盤快捷鍵（Cypress 下 app 使用記憶體剪貼簿）；複製前不可點擊空白處（會取消選取導致複製到空內容）——這些踩坑已記錄在 spec 註解。

## 充分性分析

- 表列的三個要求（web 字型、複製貼上流程、與顯示一致）皆覆蓋；搭配 Jest 層（批次轉換、revert、字型替換警告分支）此列完整。
- **真實發現（兩個高優先 bug）**：
  1. 顯示（Google Fonts 服務）與轉換（S3 TTF + fontkit）字型來源不同，直接轉換寬度比 ≈ 0.975——`docs/tests/bugs/Web字型顯示與轉路徑結果寬度差約2.5%.md`。
  2. 依 QA 要求把容差從 ±6% 收緊到已知比例 ±0.5% 後，**立刻抓到第二個 bug**：複製>原地貼上>轉路徑的結果寬度比 ≈ 0.9523 且右移 2.55%（確定性重現）——`docs/tests/bugs/複製貼上後轉路徑結果偏窄且右移.md`。這正是本列警示的回歸場景。
- 斷言策略：以「釘住已知比例 ±0.5% 窄帶」取代寬容差——測試保持綠燈，但行為任何方向的漂移（變好或變壞）都會失敗提醒；兩個 bug 修復後需依 spec 內註解把區間改回正常值。
