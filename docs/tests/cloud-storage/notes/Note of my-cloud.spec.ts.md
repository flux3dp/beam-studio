# Note of my-cloud.spec.ts

## 對應測試表項目

雲端儲存分類的八列：「確認登入後才能使用」、「免費用戶檔案上限為 5 個，儲存第 6 個檔案時跳出錯誤 pop-up」、「第一次按儲存到雲端成功儲存」、「儲存到雲端＝另存新檔（可同名）」、「儲存＝覆寫同一檔案」、「刪除、複製、重新命名功能正常」、「排序正常」、「檔案縮圖正常顯示」。

## 測試了什麼

共 9 個測試（CI 上自動略過整個檔案）：

1. 登入後開啟 My Cloud，驗證列表載入且標題顯示免費方案額度（N/5）。
2. 首次儲存：畫一個矩形後儲存到雲端，驗證清單出現正確名稱的卡片。
3. 「儲存」與「儲存到雲端」的區別：前者覆寫同一檔案（數量不變），後者建立新檔（可同名，數量加一）。
4. 重新命名後清單即時更新。
5. 複製後出現第二張卡片。
6. 刪除後清單清空。
7. 排序：建立多個可比序的檔名，切換名稱升冪／降冪，驗證卡片順序翻轉。
8. 縮圖：每張卡片有 `src` 非空的縮圖圖片（不做像素比對）。
9. 檔案上限：存滿 5 個後嘗試第 6 個，驗證出現儲存空間已滿的錯誤視窗，且數量仍為 5。

測試名稱一律使用 `cy-test-` 前綴，套件開頭有防禦性清掃、結尾有清理，確保帳號回到 0 個檔案。

## 設計理由

- 帳號密碼由 `CYPRESS_username` / `CYPRESS_password` 環境變數注入，不寫死在程式碼中，之後換正式測試帳號只需改環境變數。
- 這是會改動真實帳號狀態的測試，因此清理紀律是交付內容的一部分：每次執行前先清掉殘留檔案，執行後確認歸零，重跑不會互相污染。
- 縮圖與排序只驗證結構（圖片存在、順序翻轉），不做像素或時間戳的精確比對，避免不穩定。

## 充分性分析

- 邏輯層（API 端點、儲存與另存的分支、額度錯誤處理）已由 Jest 的 `cloudFile.spec.ts` 與 `handlers/cloud.spec.ts` 完整釘住；本 spec 補上真實後端的端對端驗證。
- **重要環境限制**：生產後端的 CSRF 信任來源只允許 `*.flux3dp.com`，因此從 localhost 執行時所有「寫入」操作（儲存／重命名／複製／刪除）都會被 403 拒絕。spec 以 `cloudWritable` 旗標偵測執行來源：在 localhost 只有測試 1（登入／列表／額度）實際執行，其餘 7 個寫入類測試自動略過並記錄原因。**要獲得完整覆蓋，需以 `cypress.config.prod.ts`（`studio.flux3dp.com`）執行，或請後端把 localhost 加入信任來源**。詳見 `docs/tests/bugs/雲端寫入操作在localhost被CSRF阻擋.md`。
- 「舊檔案開啟正常」一列未涵蓋：需要一個固定的舊格式測試檔案，見待確認問題。
