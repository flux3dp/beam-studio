# Note of framing.spec.ts

## 對應測試表項目

外框預覽分類：「外框預覽功能是否正確 — 隱藏的圖層不會進行外框預覽」。本筆記涵蓋此列的**計算邏輯層**；實機的外框動作（三種模式的開始/停止、藍光）維持人工驗證。

## 測試了什麼

共 7 個測試，皆透過公開介面 `FramingTaskManager.startFraming(FramingType.Framing)` 驅動（座標計算函式 `getCoords` 未匯出），並以傳給 `deviceMaster.startFraming` 的角點座標作為觀察點：

1. **隱藏圖層排除**：兩個可見圖層＋一個 `display="none"` 圖層，外框等於可見圖層的聯集範圍；並驗證隱藏圖層的 bbox **從未被查詢**（排除發生在收集階段，而非計算後過濾）。
2. **切回可見即擴大**：同場景把第三層改為可見，外框隨之擴大到三層聯集。
3. **排除由屬性驅動**：DOM 完全相同、僅切換 `display` 屬性，外框極值即翻轉——證明判斷依據是屬性而非元素存在與否。
4. **全部隱藏**：回傳 `false`、不呼叫裝置、顯示「無元素可預覽」訊息。
5. **`repeat=0` 排除**：可見但執行次數為 0 的圖層不納入外框。
6. **範圍夾制**：超出工作範圍的 bbox 被 clamp 到 `[0,0]`–`[300,210]`（mm）。
7. **空群組忽略**：0×0 的空 `<g>` 不影響外框。

## 設計理由

- 期望值全部手算（dpmm=10 換算為 mm），不重跑原始碼公式。
- 「隱藏圖層的 bbox 從未被查詢」這個斷言比單純比對結果更強：若日後有人把排除邏輯移到計算後段、或改壞了收集階段的判斷，此測試會立即發現。
- 裝置與 websocket 層以最薄的方式 mock（check-device-status、deviceMaster、swiftray-client），因為本列的實機動作本來就屬人工層。

## 充分性分析

- **外框（Framing）模式**：座標聯集與各項排除規則已完整覆蓋。
- **凸包（Hull）模式**：凸包由 FLUXGhost 的 `getConvexHull` 於伺服器端計算，客戶端只負責點陣化與縮放——無本地數學可測，誠實列為委派。
- **區域檢查（AreaCheck）模式**：範圍來自裝置後設資料（`exportFuncs.getMetadata`），亦非本地邏輯。
- 旋轉軸相關的 framing 路徑（RotateAxis/RotateFraming/Contour）屬旋轉軸與 Promark 裝置範疇，不在本列意圖內。
