# Note of auto-save-helper.spec.ts

## 對應測試表項目

Preference 分類：「自動儲存」（表上建議欄明示：Jest 單元測試 auto-save-helper.ts，fake timers + mock 儲存層，原本零測試）。

## 測試了什麼

共 16 個測試，覆蓋組態與計時兩大面向：

- **組態**：讀取 `auto-save-config` 儲存鍵、無組態時的預設值（啟用、5 個檔案、10 分鐘、文件夾 Beam Studio/auto-save）、init 僅在無組態時套用預設、建目錄失敗時自動改為停用、macOS 權限預寫檔案。
- **計時（fake timers）**：啟用後依設定間隔觸發儲存（驗證 .beam 路徑、內容 buffer、檔名含分頁 id）；停用不啟動；web 版為 no-op；不在編輯器頁面時跳過；重複啟動不會產生第二個計時器；停止後不再觸發；間隔變更經停止/重啟後生效；超過檔案數上限時修剪最舊的自動儲存檔（不誤刪無關檔案）；目錄消失時跳警告並略過該次儲存。

## 設計理由

- 計時器行為全部以 fake timers 驅動（afterEach 恢復 real timers 並關閉計時器，避免洩漏到其他套件）。
- 檔案系統為 mock——core 層的 `fs` 本來就是 stub，實際寫檔由 Electron 端實作，屬桌面端行為；測試以 mock 驗證呼叫合約（路徑、旗標、修剪順序）。
- 發現並釘住的合約細節：儲存有**三重閘門**（非 web、啟用且無現存計時器、位於編輯器頁）；組態變更的自動重啟僅在「有未儲存變更」時發生；刪檔失敗會重試一次後記錄並繼續（不中斷循環）。

## 充分性分析

- 自動儲存的可計算邏輯（組態、排程、修剪、異常）完整覆蓋。
- 未覆蓋：Electron 端實際寫檔與復原流程（開啟自動儲存的 .beam）——屬桌面端整合，另行人工或未來 Electron 測試考量。
