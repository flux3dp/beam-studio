# Bug（高優先）：複製 > 原地貼上 > 轉路徑的結果比直接轉路徑更窄且右移

- **狀態**：**已驗證不成立（非 bug）** — 2026-07-07 於本機 FLUXGhost rig 用對照組證明：同字串下「直接轉原件」與「貼上複本轉路徑」轉出的 path `d` 逐字元完全相同（見下方第五節）。原判定為跨不同字串比較造成的假象。**建議關閉本單、修正對應 spec 斷言。**
- ~~**嚴重性**：**高** — 同樣內容的文字，「貼上的複本」與「原件」轉出的路徑不同；貼上複本的轉換結果與畫面顯示明顯不一致（寬度 −4.8%、左緣右移 +2.55%）~~（已否證：原件與複本轉出結果 byte-identical）
- **相關檔案**：剪貼簿複製流程（貼上時的屬性序列化）、`packages/core/src/web/helpers/convertToPath.ts` 的字型/屬性讀取

## 問題描述

以 web 字型（實測 Noto Sans）建立文字後：

- **直接轉路徑**：path/text 寬度比 ≈ **0.975**（已知的字型來源差異，另一張 bug 單）。
- **複製 → 原地貼上 → 對複本轉路徑**：寬度比 ≈ **0.9523**、左緣相對原文字右移約 **2.55% 寬度**（實測 761 單位寬的文字右移 19.43 單位）。兩個數值跨執行完全一致（多次執行到小數位皆相同），屬確定性行為而非隨機誤差。

淨效果：貼上複本轉出的路徑比原件轉出的更窄、且大約「置中」而非靠左對齊——雖然貼上的文字在畫面上與原件**完全重疊**（測試已驗證四邊零位移）。也就是說，**同樣顯示內容的兩個文字元素，轉路徑結果不同**，差異來源在複製/貼上過程遺失或改變了某些影響轉換的屬性（推測方向：letter-spacing／字距、tspan 結構、或字型參照方式）。

## 重現條件

1. 建立文字（例如「TEST 123」）、套用 Noto Sans。
2. 複製（Cmd/Ctrl+C）→ 原地貼上（Shift+Cmd/Ctrl+V），確認複本與原件完全重疊。
3. 對貼上的複本執行轉路徑。
4. 量測轉出 path 的 bbox：寬度比 ≈ 0.9523、左緣右移 ≈ 2.55% 寬度（對照：直接對原件轉路徑為 0.975、零位移）。

自動化重現：`right-panel/text-to-path-web-font.spec.ts` 第 3 個測試，已把兩個數值以 ±0.5% 窄帶「釘住」——行為一旦改變（變好或變壞）測試都會失敗提醒。

## 發現經過

原測試容差為 ±6%，此問題被掩蓋；依 QA 要求把容差收緊到已知比例 ±0.5% 後立即浮現。這也是「容差不宜過寬」的實例。

## 待修復方向

1. 比對「原件」與「貼上複本」兩個 text 元素的完整屬性/子節點差異（letter-spacing、tspan、data-* 等）。
2. 修復後，測試中的 `KNOWN_PASTED_CONVERT_*` 兩組常數應改回與直接轉換相同的區間（spec 內有註解指引）。

## 驗證記錄（2026-07-07）

### 一、轉路徑引擎確認：**2.0（fontkit，前端本地轉換）**

與姊妹 bug（`Web字型顯示與轉路徑結果寬度差約2.5%.md`）相同，判定關鍵在 `packages/core/src/web/app/actions/beambox/font-funcs.ts:710`：

```ts
let preferGhost = !pathPerChar && globalPreference['font-convert'] === '1.0';
```

- 預設 `font-convert` = `'2.0'`（`beambox-preference.ts:42`）、面板轉路徑 `pathPerChar=false`、Noto Sans 為 LTR → `preferGhost=false`。
- 執行 `convertTextToPathByFontkit(...) || convertTextToPathByGhost(...)`（`font-funcs.ts:746-748`），fontkit 成功回傳，**兩件（原件與貼上複本）都由 fontkit 2.0 轉出**。故「原件 0.975 vs 複本 0.9523」是**同一引擎、同一字檔**下的差異，變因只能來自兩個 `<text>` DOM 的差別，而非引擎切換。

### 二、機制：偏差來自 `getStartPositionOfChar`（DOM 排版），故對 DOM 差異敏感

fontkit 分支對每個字用 `char.path.scale(sizeRatio).translate(x, y)` 定位，其中 `x,y = tspan.getStartPositionOfChar(i)`（`font-funcs.ts:351-359`）——這是**瀏覽器對該 text 元素當下排版**算出的座標。因此只要貼上複本的 DOM（tspan 結構 / `x` 座標清單 / letter-spacing / 當下套用的字型度量）與原件不同，轉出的字元位置就整體偏移、末字右緣內縮 → 觀察到的「更窄 0.9523 + 右移 2.55%」。

- **跨執行位元完全一致（小數位皆同）** 這點很關鍵：若是「貼上時 web font 尚未套到複本、被 fallback 度量量到」的載入競速，結果應會浮動；穩定重現反而指向**結構/屬性層面的確定性差異**（貼上序列化改寫了影響 `getStartPositionOfChar` 的屬性，例如顯式 per-char `x` 清單、letter-spacing 或 tspan 拆分）。→ 優先往剪貼簿序列化查，而非往時序 race 查。

### 三、詳細操作 → 對應程式碼

| 步驟 | 操作 | 進入的程式碼 |
|---|---|---|
| 1 | Text → 輸入 `Paste42` → 選 Noto Sans | 建立 `<text id=svg_1>` |
| 2 | `Cmd/Ctrl+C` | 複製（Cypress 下為 MemoryClipboard，見 NativeClipboard） |
| 3 | `Shift+Cmd/Ctrl+V` 原地貼上 | 新增第二個 `<text>`（複本），與原件四邊零位移重疊 |
| 4 | 對複本按「轉路徑」`#to_path` | `fontFuncs.convertTextToPath`（`font-funcs.ts:644`）→ `convertTextToPathByFontkit`（`font-funcs.ts:223`） |
| 5 | 結果 | 複本 `<text>` → `<path>`；原件 `<text>` 保留 |

### 四、已實測結果（跨執行確定性，來自 `text-to-path-web-font.spec.ts` 第 3 測）

| 對象 | path/text 寬度比 | 左緣位移 |
|---|---|---|
| 直接對原件轉路徑 | 0.975 | ≈0 |
| 複製→原地貼上→對複本轉路徑 | **0.9523** | **右移 ≈2.55% 寬**（761 單位寬文字右移 19.43 單位） |

### 五、決定性實驗（本機 FLUXGhost rig 現場擷取，2026-07-07）：**此 bug 不成立**

方法：用臨時 Cypress 擷取 spec（調查後已移除）於 rig 上跑一個**對照組**——把「原件 `Paste42`」
**不經複製貼上、直接轉路徑**，與「貼上複本轉路徑」用**同一字串**逐一比對。

實測（`Paste42`、Noto Sans、font-size 200、2.0 fontkit）：

| 對象 | path/text 寬度比 | 左緣位移比 | 轉出 path `d` |
|---|---|---|---|
| **直接轉原件**（無貼上） | **0.95231** | **0.02616** | 3824 字元 |
| **複製→原地貼上→轉複本** | **0.95231** | **0.02616** | 3824 字元 |

**兩者的 path `d` 逐字元完全相同（byte-identical，3824 字元一致）、bbox 一致、寬度比與位移比到小數第五位皆同。**

原件與貼上複本的 `<text>` DOM 也僅有無關排版的差異：貼上複本少了 `style="pointer-events:inherit"`、tspan 多一個 `id`；`x`/`y`/`font-size`/`font-postscript`/tspan 座標全同，兩個 text bbox 完全重疊（742.77 寬、同 x）。

**結論：0.9523 的寬度比與 2.6% 的左緣位移，是字串 `Paste42` 在 Noto Sans + 2.0 fontkit 下的固有特性，與「複製貼上」無關。** 原始 bug 單把「貼上 `Paste42`→0.9523」拿去對照「直接 `Web123`→0.975」——那是**兩個不同字串**（不同字元、不同左側邊距），比值本來就不同；差異全部來自字串差異，而非貼上。之前 ±6% 容差被收緊後看到的「複本更窄、右移」是**跨字串比較的假象**。

gcode：直接組與貼上組的匯出 gcode 僅有 ≤0.1mm 的 X 掃描端點量化抖動（約 178/10127 行，所有 Y 完全相同），屬 raster 量化捨入雜訊，**非**幾何差異；遠不是原單所述的「窄 4.8%／右移 2.55%」。（擷取用臨時 spec 與輸出檔已於調查後移除。）

> 註：`Paste42` 直接轉路徑的 0.9523 本身，同樣受 [[Web字型顯示與轉路徑結果寬度差約2.5%]] 的字源不一致（S3 舊 build）＋ ink/advance 量測差影響，與「貼上」無關。

### 六、對測試與 bug 單的處置建議

1. **本 bug 單降級／關閉**：「複製貼上改變轉路徑結果」不成立（同字串轉出 `d` 逐字元相同）。
2. **修正 spec**：`text-to-path-web-font.spec.ts` 第 3 測目前把 `KNOWN_PASTED_CONVERT_*` 釘在 0.9523，語意上被當成「貼上造成的回歸」，實際上它量到的是 `Paste42` 這個字串的固有比值。正確的「複製貼上不改變轉換」斷言應改為：**同字串下，貼上複本轉出的 `d`（或寬度比）＝直接轉原件**（此為真實不變量，會通過）。
3. 唯一殘留的真實議題是 [[Web字型顯示與轉路徑結果寬度差約2.5%]]（字源不一致），與貼上無關。
