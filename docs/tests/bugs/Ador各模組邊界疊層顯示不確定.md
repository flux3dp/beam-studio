# Bug（潛在）：Ador 各雷射/列印模組的工作範圍邊界顯示，受先前切換過的模組影響（不具確定性）

- **狀態**：未解（需產品確認 `use-union-boundary` 預設行為是否符合預期）
- **嚴重性**：中 — 邊界顯示與模組切換「順序」相關，同一模組在不同操作路徑下可能看到不同邊界
- **檔案**：`packages/core/src/web/app/svgedit/...` 邊界繪製（`boundaryDrawer.ts`）與 `use-union-boundary` 偏好設定

## 問題描述

Ador 各模組的實體加工範圍（例如 10W 430×300、20W 430×290、列印 430×270、2W 紅外 430×282）不是反映在畫布 `#svgcontent` 的 viewBox（固定 430×320），而是畫成獨立的 `#boundary-path` 疊層。在預設開啟的 `use-union-boundary` 之下，這個疊層會「合併曾經切換過的模組的邊界偏移」。

實測結果：同一個 10W 模組的邊界 path，在一種切換順序下量到 `V3000`，換一種順序卻量到 `V2700`——也就是邊界顯示依賴歷史狀態，不是模組的純函數。

## 重現條件

1. 工作範圍設為 Ador，依序切換模組：列印 → 10W → 檢視 `#boundary-path` 的 `d` 值。
2. 重新整理後改用另一種順序（例如直接選 10W）再檢視 `d` 值。
3. 兩者不一致。

## 影響範圍

- 使用者看到的可加工範圍提示可能與當前模組實際範圍不符。
- 自動化測試無法對 430×270/290/300/282 這幾列做穩定斷言（`workarea-modules.spec.ts` 因此僅驗證模組選取狀態與畫布 viewBox，已於測試註解與 note 說明）。

## 待產品確認

- `use-union-boundary` 的預設開啟是否刻意（例如避免使用者切換模組時物件掉出範圍）？
- 若邊界顯示應為「當前模組」的確定函數，修正後即可補上這四列的數值斷言。
