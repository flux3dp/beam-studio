# Note of auto-shrink.spec.ts

## 對應測試表項目

文件設定分類：「測試自動內縮功能是否有正確作用（用 HEXA 測試複合填充路徑）」。實機雕刻結果維持人工抽查；本 spec 覆蓋自動內縮對**產生的工具路徑**的實際作用。

## 測試了什麼

共 3 個測試（需 FLUXGhost，CI 自動略過；場景：HEXA 工作範圍、40×40mm 外方減去同心 20×20mm 內方以布林差集做成帶孔複合路徑、填充雕刻、500 DPI）：

1. **覆蓋度量（OFF vs ON）**：以逐掃描線座標比對——外框邊緣向內縮 0.05mm；內孔兩側各向外擴 0.05mm（孔寬 +0.10mm、垂直方向也各多出掃描線）；內部填充仍存在（>95% 保留、雕刻總長度僅 −0.67%）；內縮量在外框與內孔各邊**一致**（0.05mm = HEXA 的 autoShrink 常數 = 500 DPI 下恰好 1 pixel）。
2. **Golden gcode**：OFF 與 ON 兩份固定場景基準檔逐位元比對（`golden-gcode/auto-shrink-hexa-compound-500dpi-{off,on}.gc`）。
3. **<250 DPI 閘門**：125 DPI 時開關 ON/OFF 產生**逐位元相同**的 gcode——驗證 tooltip 所述的 DPI 門檻確實生效。

## 設計理由

- 「自動內縮難以用尺寸差驗證」的問題，以**逐掃描線座標**解法：從 gcode 的精確座標計算每條掃描線的起迄與內孔間隙，等同確定性的視覺驗證，不需要真實渲染、無像素比對的不穩定性。
- 選 500 DPI 而非門檻值 250：0.05mm 的 erode 在 250 DPI（dpmm=10）只有半個 pixel，效果不對稱且有雜訊；500 DPI 下恰為 1 pixel，每條邊都乾淨落點，斷言穩健（門檻 ≥250 仍滿足）。
- 兩層驗證依 e2e-test.md 的路徑驗證準則；共用模組 `taskPath.ts` 未修改（孔隙需要的逐線多段擷取屬場景特定，留在 spec 內）。

## 充分性分析

- 自動內縮的可計算合約（內縮方向、幅度、均勻性、DPI 閘門、對複合路徑內外邊界的對稱作用）完整覆蓋，並受 golden 保護。
- 發現一個實作細節：客戶端**不論 DPI 都送出** `engraving_erode` 參數（`svg-laser-parser.ts:426-435`），門檻是在 FLUXGhost 引擎內部執行的——行為正確，但責任分工在文件上並不明顯。
- 未覆蓋：實機雕刻的實際內縮效果（材質、光斑交互）——人工。

## 本次改進

新增本 spec 與兩份 golden 基準檔。

## 待確認問題（Open Questions）

1. **250 DPI 恰好在門檻上時的次像素行為**：0.05mm erode 在 250 DPI 是半個 pixel，實測效果不對稱（只有部分邊緣移動 0.1mm）。tooltip 說「≥250 DPI 適用」，但在門檻值上的效果並不完整——是否應把門檻改為 >250、或在 UI 提示 500 DPI 以上效果最佳？
2. HEXA 的內縮常數 0.05mm 是否有規格出處？其他支援機型（若有）的常數是否不同、需要各自的測試場景？
